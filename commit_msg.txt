fix: 重写点赞/收藏逻辑,消除嵌套 Promise 与提前解锁

根因:
- 旧 addLike 在有孤儿记录时串联 get + remove*多 + add + inc,
  3~5 个云调用排队累加,触发云开发 SDK 内部 timeout
- 旧 likeLock 用 setTimeout(800ms) 提前释放,云调用未完成锁就开了,
  用户连点导致并发执行、状态错乱
- removeOps = res.data.map(item => ...remove()) 这种
  map+Promise.all 的嵌套,错误容易在链中间丢失

修法:
- addLike: 纯串行 2 步:add 一条 + inc(+1),不再查重清理
- cancelLike: 纯串行 3 步:get(limit 1) → doc.remove → inc(-1)
- addFavorite/removeFavorite: 同样精简到 2~3 步串行
- likeLock 改成 .finally() 自动释放,云调用完才解锁
- 数字变化统一通过 inc 由云端维护,UI 用本地 state 反映
- 同步修复 detail.js 和 favorites.js 的一致性问题

行为: 点一下 → 红心+1;再点一下 → 白心-1,可靠
